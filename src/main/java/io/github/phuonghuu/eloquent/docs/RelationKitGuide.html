RelationKit

A lightweight Eloquent-style relation kit for Java. It is not an ORM replacement. The goal is explicit SQL, batch loading, eager loading, and relation metadata with no Hibernate/JPA dependency.

Java 17+ Spring Boot 3+ PostgreSQL NamedParameterJdbcTemplate

Setup

Register entities and relations by code inside RelationRegistry, then inject RelationKit and use relationKit.query(Entity.class). 

registry.forClass(Department.class)
    .table("comission.departments")
    .primaryKey("id")
    .belongsToMany("users", User.class, "comission.positions", "department_id", "user_id", Department::setUsers)
    .hasOneThroughPivot(
        "manager",
        User.class,
        "comission.positions",
        "department_id",
        "user_id",
        "role_code",
        "MANAGER",
        Department::setManager
    )
    .register();
Use code registration first. No annotations are required for the current version.

Query Builder

Start a query with relationKit.query(Type.class). Every method returns the same builder so you can chain filters, projections, ordering, relations, and paging.

select(...)

Pick specific columns on the root query or inside relation callbacks.

relationKit.query(Department.class)
    .select("id", "code", "name")
    .get();

where(...) / orWhere(...)

Support equality and explicit operators. Nullable values resolve to IS NULL.

.where("status", "ACTIVE")
.where("id", "<", 10)
.orWhere("level", ">=", 2)

whereRaw(...)

Safe raw predicate with ? placeholders and positional binding.

.whereRaw("level_no > ? AND status = ?", List.of(3, "ACTIVE"))

whereNull / whereNotNull

Null checks without manual SQL strings.

.whereNull("description")
.whereNotNull("code")

whereBetween

Range filter with two bound values.

.whereBetween("id", 1L, 100L)

whereLike

LIKE predicate for pattern search.

.whereLike("name", "%Dept%")

whereDate

Day-level filtering for date/time columns.

.whereDate("createdAt", LocalDate.of(2026, 5, 31))

whereJson

PostgreSQL JSON containment filter.

.whereJson("meta", Map.of("kind", "HQ"))

whereIn

Use for batch filtering. Empty collections short-circuit to no rows.

.whereIn("id", ids)

whereHas / whereHasIn

Filter the parent based on child relation existence.

.whereHas("users", q -> q.where("status", "ACTIVE"))
.whereHasIn("users", "id", List.of(10L, 11L))

orderBy

Explicit order direction is supported and recommended.

.orderBy("id", "ASC")
.orderByDesc("id")

page / size / limit / offset

Offset paging for normal queries and relation-local limits.

.page(1).size(20)
.limit(10)
.offset(0)

Relation Types

  • hasOne: parent to one child via local key and child key.
  • hasMany: parent to many children via local key and child foreign key.
  • belongsTo: child points to one parent via foreign key.
  • belongsToMany: many-to-many through pivot table.
  • hasOneThroughPivot: one related row selected through pivot with an extra pivot filter.
registry.forClass(User.class)
    .table("comission.users")
    .primaryKey("id")
    .hasMany("orders", Order.class, User::getId, Order::getUserId, User::setOrders)
    .register();

Eager Loading

Use with(...) to batch-load relations. Nested paths are supported, for example with("users.orders"). 

relationKit.query(Department.class)
    .with("users", q -> q.select("id", "username").orderBy("id").limitPerParent(3))
    .with("users.orders", q -> q.select("id", "amount", "user_id"))
    .with("manager")
    .get();
Relation callbacks can reuse select, where, orderBy, limit, and the other filtering methods. This keeps child SQL explicit and batch-friendly.

Write / Mutation

Mutation APIs live under relationKit.write(Entity.class). This keeps the read path clean and makes side effects explicit. The current write facade focuses on plain inserts, updates, upsert-by-primary-key, and pivot mutation for belongsToMany.

create(...) / createMany(...)

Insert one entity or a collection of entities using the registered table metadata.

RelationWriter<User> writer = relationKit.write(User.class);

writer.create(User.builder()
    .id(100L)
    .username("alice")
    .status(StatusEnum.UserStatus.ACTIVE)
    .createdAt(ZonedDateTime.now())
    .updatedAt(ZonedDateTime.now())
    .build());

writer.createMany(List.of(
    User.builder().id(101L).username("bob").status(StatusEnum.UserStatus.ACTIVE).build(),
    User.builder().id(102L).username("carol").status(StatusEnum.UserStatus.ACTIVE).build()
));

update(...) / updatePartial(...) / updatePartialByMap(...) / updatePartialByMap(id, ...) / upsert(...)

update writes all simple fields. updatePartial only touches the fields you list. updatePartialByMap is useful when the patch comes from a DTO or request map. The (id, patch) overload updates directly without loading the entity first. upsert updates first, then inserts if the row is missing.

update(...)

RelationWriter<User> writer = relationKit.write(User.class);

User user = relationKit.query(User.class).find(1L);
user.setUsername("alice.updated");
user.setStatus("ACTIVE");
writer.update(user);

updatePartial(...)

RelationWriter<User> writer = relationKit.write(User.class);

User user = relationKit.query(User.class).find(1L);
user.setUsername("alice.patched");
user.setStatus(null);
writer.updatePartial(user, "username", "status");

updatePartialByMap(...) / updatePartialByMap(id, ...)

RelationWriter<User> writer = relationKit.write(User.class);

Map<String, Object> patch = new LinkedHashMap<>();
patch.put("username", "alice.map");
patch.put("updated_at", null);

User user = relationKit.query(User.class).find(1L);
writer.updatePartialByMap(user, patch);
relationKit.updatePartialByMap(User.class, 1L, patch);

upsert(...)

RelationWriter<User> writer = relationKit.write(User.class);

User newUser = new User();
newUser.setId(200L);
newUser.setUsername("new-user");
newUser.setStatus(StatusEnum.UserStatus.ACTIVE);
writer.upsert(newUser);
Use updatePartial(...) when you want a patch-style write. Fields you do not list stay untouched, while listed fields can still be set to null explicitly. Use updatePartialByMap(...) when you already have a key/value patch object from API input. Use the (id, patch) overload when you want to patch directly without loading the entity first.

attach(...) / detach(...) / dettach(...)

Attach or remove rows in a many-to-many pivot table. Current support targets belongsToMany.

RelationWriter<User> writer = relationKit.write(User.class);

writer.attach("roles", 1L, List.of(10L, 11L));
writer.detach("roles", 1L, List.of(10L));

// Alias kept for backward compatibility with a typo in older code
writer.dettach("roles", 1L, List.of(11L));
If your pivot table has extra columns, attach or sync a per-id map of pivot attributes: 
writer.attach("users", departmentId, Map.of(
    15L, Map.of("role_code", "STAFF", "level_no", 1),
    16L, Map.of("role_code", "MANAGER", "level_no", 2)
));

writer.sync("users", departmentId, Map.of(
    15L, Map.of("role_code", "STAFF", "level_no", 1),
    16L, Map.of("role_code", "MANAGER", "level_no", 2)
));

writer.syncWithoutDetaching("users", departmentId, Map.of(
    17L, Map.of("role_code", "VIEWER", "level_no", 3)
));

sync(...) / syncWithoutDetaching(...)

sync replaces the current pivot set with the provided ids. syncWithoutDetaching only adds missing ids and keeps existing links. 

RelationWriter<User> writer = relationKit.write(User.class);

writer.sync("roles", 1L, List.of(10L, 12L));
writer.syncWithoutDetaching("roles", 1L, List.of(13L, 14L));

// Backward-compatible alias
writer.syncWithoutDetachting("roles", 1L, List.of(15L));

previewAttachSql(...) / previewDetachSql(...) / previewSyncSql(...)

These helpers return compact SQL previews for pivot writes. They are useful when you want to log the SQL before mutating data, for example in controllers or benchmark endpoints. 

RelationWriter<User> writer = relationKit.write(User.class);

log.info("attach preview: {}", writer.previewAttachSql("roles", 1L, List.of(10L, 11L)));
log.info("detach preview: {}", writer.previewDetachSql("roles", 1L, List.of(10L)));

log.info("sync preview: {}", writer.previewSyncSql("users", departmentId, Map.of(
    15L, Map.of("role_code", "STAFF", "level_no", 1),
    16L, Map.of("role_code", "MANAGER", "level_no", 2)
)));
Mutation APIs currently work on belongsToMany relations for pivot operations. Root create, update, and upsert work on any registered entity table.

Terminal Methods

first()

Returns the first row for the current query state, or null when empty.

Department department = relationKit.query(Department.class)
    .where("status", "ACTIVE")
    .orderByDesc("id")
    .first();

find(id)

Shortcut for primary-key lookup using the registered metadata.

Department department = relationKit.query(Department.class)
    .with("users")
    .find(1L);

Pagination

paginate(page, size)

Offset pagination with metadata: total rows, total pages, has next, has previous.

PaginatedResult<Department> page = relationKit.query(Department.class)
    .where("status", "ACTIVE")
    .orderBy("id", "ASC")
    .paginate(1, 20);

cursorPaginate(lastId, size)

Keyset pagination optimized for large tables. Cursor is the primary key by design.

CursorPaginatedResult<User> page = relationKit.query(User.class)
    .orderBy("id", "DESC")
    .cursorPaginate(null, 20);
For cursor pagination, the current implementation supports ordering by the primary key only. Use null for the first page. Then pass nextCursorId() into the next request.

SQL Preview

Use these methods to inspect generated SQL in logs or during debugging.

  • toSql() - root query only
  • toDebugSql() - root query with literal values
  • toSqls() - root + eager load SQL list
  • toDebugSqls() - compact debug SQL list for ELK or logs
List<String> sqls = relationKit.query(Department.class)
    .select("id", "code")
    .with("users", q -> q.select("id", "username"))
    .toDebugSqls();

Practical Example

List<Department> departments = relationKit.query(Department.class)
    .select("id", "code", "name")
    .where("id", ">", 0L)
    .with("users", q -> q
        .select("id", "username", "fullname")
        .orderBy("id", "ASC")
        .limitPerParent(3))
    .with("manager")
    .orderBy("id", "DESC")
    .cursorPaginate(null, 20)
    .items();

This keeps the SQL explicit, batch-loaded, and predictable for debugging and benchmarking.
